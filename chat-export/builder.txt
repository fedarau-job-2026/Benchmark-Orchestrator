     Benchmark Orchestrator — Implementation Plan

     Context

     FDE take-home for H Company (docs/ASSIGNMENT.md): build (1) a rate-limited
     HTTP inference service wrapping Ollama, and (2) a benchmark orchestrator
     that discovers the service's rate limits on its own and pushes a
     1,000-query job queue through it fast, reliably, and observably. Reviewers
     will re-run with different RPM/concurrency values, so nothing may be
     hard-coded. Heaviest evaluation weight: concurrency & backoff behavior,
     then system design, observability, product feel, pragmatism.

     Decisions made with user:
     - All Python. Two fully separate, non-crossed deployments simulating real 
     life: inference service and orchestrator are independent projects — own
     pyproject, own lockfile, own Dockerfile, own tests, zero shared code. The
     only contract between them is HTTP. Compose at the repo root ties them
     together for convenience, but each builds and runs standalone.
     - Build order: start with the system under test (inference service) and its
     tests; finish with our probe (the orchestrator).
     - Ollama runs as a third compose container by default, with OLLAMA_URL env
     override to point at a host install; service also ships a mock backend for
     dev/review without a model.
     - Rate-limit discovery: adaptive AIMD controller (no configured limits,
     continuous adaptation).
     - Observability: Rich live TUI. Native uv run path must work alongside
     Docker.
     - Decision records: every significant decision gets aggregated per
     subproject in a dedicated decisions/ folder (ADR-style), written to expose
     our thinking process to the LLM/human reviewers who will evaluate the
     result. Cross-cutting decisions live in a root-level decisions/ folder.
     - CLAUDE.md at repo root codifying all of the above as standing rules for
     any future session working in this repo.

     Machine facts: Docker 29.6.1 daemon running, but no docker compose plugin →
     brew install docker-compose + ~/.docker/config.json cliPlugins entry
     needed. No Ollama installed. System Python is 3.9 → uv-managed 3.12.
     Existing repo: docs/, data/benchmark.csv (100 rows), data/queue.jsonl (10
     runs), .gitignore (already ignores results/, .venv/).

     Component design

     Inference service (deployment #1 — what we'll be testing)

     - POST /generate {"prompt": str} → {"response": str, "model": str}; GET 
     /healthz.
     - Contract shared via Swagger/OpenAPI: FastAPI serves interactive Swagger
     UI at /docs and the machine-readable spec at /openapi.json, with the 429
     responses, Retry-After/X-RateLimit-Reason headers, and all schemas
     explicitly declared (Pydantic models + responses= annotations). The spec is
     exported to docs/openapi.json — that committed artifact IS the contract
     the orchestrator codes against (no shared Python code). A service-side
     contract test asserts the live /openapi.json matches the committed spec so
     drift is caught.
     - Rate limiter (built by hand per ground rules):
       - RPM: deque of accepted-request timestamps within the trailing 60 s;
     reject when full. Rejected requests do NOT consume window slots (documented
     choice).
       - Concurrency: atomic in-flight counter; reject when at cap.
       - On breach: 429 + Retry-After header (RPM: ceil of seconds until the
     oldest timestamp exits the window; concurrency: 1), plus advisory
     X-RateLimit-Reason: rpm|concurrency — the orchestrator treats it as an
     optional hint only.
       - No internal queueing; check-and-reject before touching the backend.
     - Backends: OllamaBackend (httpx → {OLLAMA_URL}/api/generate, keep_alive,
     model from env) and MockBackend (asyncio.sleep, configurable mean/jitter
     latency, canned response; accuracy meaningless in mock mode — documented).
     - Env config: RPM=60, MAX_CONCURRENCY=4, BACKEND=ollama|mock, OLLAMA_URL,
     MODEL=qwen2.5:0.5b, MOCK_LATENCY_MS, PORT=8080.
     - Its tests (written with/before the implementation): RPM window edges
     (61st request in a minute → 429; slot frees exactly when oldest ages out),
     concurrency cap under parallel load, correct Retry-After values,
     no-queueing behavior, mock-backend latency, config parsing. Uses FastAPI
     test client + a fake clock for the limiter.

     Orchestrator (deployment #2 — our probe, built last)

     - Jobs: queue JSONL → JobSpec (with type defaulting to benchmark_csv);
     registry maps type → (task expander, grader). CSV job expands to 100 query
     tasks. New job type = one registry entry, satisfying the extensibility
     criterion.
     - AIMD controller (controller.py) — fully self-tuning, no configured
     limits:
       - Concurrency window: additive increase (+1 after N consecutive clean
     completions), multiplicative decrease (×0.5, floor 1) on
     concurrency-flavored 429s.
       - RPM pacer: token bucket whose refill rate is learned — starts
     optimistic-but-polite, decreases on rate-flavored 429s (observed 429 timing
     + Retry-After), creeps back up during sustained clean periods so a mid-run
     RPM doubling gets exploited and a halving gets respected within seconds.
       - Without the reason hint header, adapts conservatively on both knobs.
     - Client: 429 → wait Retry-After + jitter and retry (429 is expected
     traffic; only a failure after a generous cap); timeouts/5xx → exponential
     backoff, ~3 attempts, then recorded as failure. Every attempt feeds the
     controller.
     - Metrics/report: per-request latency samples → p50/p95; total &
     per-benchmark wall time; throughput; total requests (incl. retries) vs
     completed; 429 count; failures; overall + per-benchmark accuracy; final
     controller estimates → results/results_<runid>.json (+ per-query detail
     CSV).
     - TUI: Rich Live — overall + per-benchmark progress bars with ETA, live
     req/s, p50/p95, in-flight vs current window, learned RPM estimate,
     429/failure counters, recent-events tail. Falls back to periodic plain log
     lines when not a TTY or --no-tui.
     - Its tests: controller unit tests (increase/decrease trajectories,
     Retry-After honoring, recovery after limit change — driven by a scripted
     fake server), jobs/grading/report units, and an integration test that boots
     the real inference service in mock mode as a subprocess (black-box over
     HTTP, keeping deployments non-crossed) and runs a small queue end-to-end
     asserting every required report field.

     Docker / compose

     - ollama: ollama/ollama image, named model volume, entrypoint wrapper pulls
     qwen2.5:0.5b on first start, healthcheck.
     - inference: builds inference-service/Dockerfile; depends_on: ollama

     - Native path: uv run inference-service / uv run orchestrator
